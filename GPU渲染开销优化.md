# GPU渲染开销优化



## 带宽优化

### 带宽

带宽是衡量显卡性能的重要指标，它反映了显存数据传输的能力（IO）

**显存位宽：**显存在一个时钟周期传输数据的位数;

**显存频率：**显存的速度，为时钟周期的倒数；

**显存带宽：**显存频率 * 显存位宽 / 8, 单位为字节每秒；

数据的频繁传输非常耗电，因此带宽是影响手机发热的重要因素之一。

### 影响带宽的操作

带宽的占用和显存读写密切关联，从类型上来分，包括CPU内存和GPU显存的交互，以及GPU内部显存的读写。一些可能和带宽相关的操作包含：

1. CPU向GPU提交命令，发送数据
2. 纹理采样 / 缓冲区读取数据
3. 颜色/深度数据写入Render Target或FrameBuffer
4. 深度测试（Alpha Test）阶段和混合（Blending）阶段读写FrameBuffer
5. 纹理大小过大，采样写入频繁
6. OverDraw严重，导致同一像素位置被写入多次

### 架构和带宽

#### 延迟管线与带宽

到目前为止，市面上绝大部分移动端渲染都是前向管线。这是因为延迟管线一般都会写入至少3个RT的GBuffer，这会带来非常严重的性能消耗。

####  后处理与带宽

后处理包括MSAA，Bloom，DOF等技术，需要对FrameBuffer进行读写。我们经常会说后处理会对性能造成影响，一方面是因为它要在一帧内触发多次framebuffer的切换，另一方面是它要频繁读写framebuffer。

## 纹理优化

 纹理是GPU编程中最常见的资源类型之一，最符合直觉的优化方式就是降低纹理大小。除此之外还有如下优化方案：

### 纹理压缩

对纹理进行压缩处理。对于美术纹理，一般会使用常见的几个块状压缩算法，通过CPU压缩，并在GPU中解压并读取，因为是按块压缩，所以解压时也只需要按所需块解压即可。这通常需要硬件支持解压。同时，硬件可能也会支持FrameBuffer的自动压缩。

#### 各平台推荐的纹理压缩格式

##### PC端

对于配备 DirectX 11 或更高级别 GPU 的设备，因其可保证支持 BC7 和 BC6H 格式，推荐的**压缩**格式选择如下：RGB 纹理使用 DXT1，每像素四位。RGBA 纹理使用 BC7（质量更高，但压缩速度较慢）或 DXT5（压缩速度更快），两者均为每像素八位。HDR 纹理使用 BC6H，每像素八位。

对于配置低于DirectX 11的设备则应优先选择 DXT5 而不是 BC7，因为这些 GPU 不支持 BC7 或 BC6H。

##### iOS

对于使用 A8 芯片（2014）或更高版本的 Apple 设备，ASTC 是 RGB 和 RGBA 纹理推荐使用的**纹理格式** 。如果需要支持较旧的设备则推荐使用 ETC/ETC2

##### Android

对于 LDR RGB 和 RGBA 纹理，大多数支持 OpenGL ES 3.1 或 Vulkan 的现代 Android GPU 也支持 ASTC 格式，如果你需要支持较旧的设备则推荐使用 ETC/ETC2

> **注意：**对于不支持 ASTC 格式的较旧 Android 设备，纹理会被解压为未压缩的 32 位 RGBA 格式。这会增加内存占用，并可能降低在 CPU 上执行解压的速度。

### mipmap

mipmap本身是为了解决纹理采样中出现走样问题的。它在原纹理的基础上，生成一系列纹理，每个纹理是前一个纹理大小的1/2，采样纹理时会选择合适大小的纹理进行采样。

#### mipmap流式加载

开启mipmap后，对于camera视图中的每个纹理，Unity会自动仅计算并加载到某个特定级别的mipmap级别，而不是加载全部级别。带宽压力上会减小，因为原本需要传输高精度纹理，现在只需要传输低精度纹理

##### 限制

- Unity 不支持对 terrain 纹理进行 mipmap 流送，因为 Unity 需要始终使用最高分辨率的 mipmap 级别。
- Unity 不支持将 mipmap 流式传输用于纹理数组、 **cubemap** 数组或 3D 纹理。

### 纹理过滤

不同的纹理过滤模式会执行不同次数的纹理采样。如下所示

| 纹理过滤         | 采样次数 |
| ---------------- | -------- |
| 最邻近采样       | 1        |
| 双线性采样       | 4        |
| 三线性采样       | 8        |
| 各向异性采样 1X  | 8        |
| 各向异性采样 2X  | 16       |
| 各向异性采样 4X  | 32       |
| 各向异性采样 8X  | 64       |
| 各向异性采样 16X | 128      |

因此如果非必要，尽量使用更低采样次数的过滤模式

### 稀疏纹理

针对整体大到无法完整装入显存的纹理。Unity会将主纹理拆分为更小的矩形区域，即”瓦片“。随后即可根据需要加载单独的瓦片。稀疏纹理在使用上与其他任何纹理的行为都相同。 **Shaders**  无需进行特殊修改即可使用它们，并且它们可以具有 mipmap、使用所有纹理过滤模式等

#### 限制

* 在 DirectX 系统上，它们需要 DX11.2（Windows 8.1）以及较新的 GPU。
* 在 OpenGL 上，它们需要支持 ARB_sparse_texture 扩展。
* 稀疏纹理仅支持未压缩的 **texture formats**

### 流式虚拟纹理

**虚拟纹理**这个概念取自Virtual Memory，与虚拟内存类似的是，一个很大的Texture将不会全部加载到内存中，而是根据实际需求，将需要的部分加载。它对基于分块压缩算法格式的贴图有很好的支持。基本思路是会将纹理的mipmap链分割为相同大小的tile，然后通过某种映射，映射到一张内存中存在的纹理，当视角变化时，一部分物理纹理会被替换，一部分物理纹理被加载

![img](https://picx.zhimg.com/v2-8dd6e9e7750884bbab8d6da5fe221ec3_1440w.jpg)

**流式虚拟纹理**是一项可减少 GPU 内存占用和纹理加载时间的技术，它基于虚拟纹理将物理纹理拆分成图块，并在需要时逐步将图块上传到GPU内存。

## OverDraw优化

对于不透明物体的OverDraw主要是注意排序问题，如果不透明严格按照从前往后渲染的排序，就不太容易出现OverDraw问题。同时由于硬件本身具有Early-Z技术，能够避免像素的重复计算。

### Alpha Test的OverDraw

Alpha Test本质上和不透明物体一样，解决排序问题可避免OverDraw问题，但由于Alpha Test的物体会在片元着色器里做discard操作，会导致Early-Z无法执行（因为执行片元着色前无法保证片元不会子被丢弃)。而导致无法避免像素的重复计算。

为了解决这个问题，一般会在渲染Alpha Test物体之前执行一次Depth-Only Pass，我们称为Pre Depth Pass。然后在Shading Pass不使用Discard操作来应用Early-Z解决像素重复计算问题。

### Alpha Blend的overdraw

由于半透明物体通常不写入深度和深度测试。所以容易出现非常严重的OverDraw。通常的解决方案有：

* 采用半分辨率离屏渲染半透明物体。在Final Pass混合Camera RT到FrameBuffer
* 采用抖动（Dithering）技术，用Alpha Test代替Alpha Blend实现半透明效果。

### 顶点密度导致OverDraw

如果物体未设置合理LOD，导致物体离相机较远时，屏幕空间的顶点密度过高，可能有多个顶点落在同一个像素，导致overdraw。

## Shader优化

### ALU指令数量优化

针对着色器计算过于复杂的情况，可以考虑如下优化建议：

1. 如果不需要准确的结果，可以用近似公式来代替计算
2. 如果半浮点数就已经满足计算的精度/范围，使用半浮点数来替代浮点数；
3. 使用查找表/纹理来代替复杂的计算；
4. 尽量避免int和float的混合计算，直接使用float计算，因为int到float的转换会消耗一定性能；
5. 考虑逐顶点计算取代逐像素；
